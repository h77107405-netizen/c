We cite official docs and best practices for each technology (Azure, Vercel, Cloudinary, Cloudflare, PWA, etc.) to guide every step.

Phase 0 – Planning & Data DesignDefine Roles & Scope: Early on, clearly define roles (Student, Teacher, Admin) and their permissions. Roles should map to access rules in your backend (RBAC). For example, only Admin can manage users/courses; Teachers can upload content and view student results; Students can study materials and take tests. This prevents design confusion later.Data Model: Plan your database schema. Common entities include Users (with fields like name, email, role_id), Courses, Subjects, Chapters, Batches, Materials, Tests/Questions, Results, Assignments, Doubts, Fees/Payments, Notifications, etc. Use an RBAC schema: store roles and permissions in tables rather than hard-coding. This allows updating roles without code changes. For simplicity, you may store each user’s role directly on the user record (e.g. role = "teacher" or role_id).Naming Conventions: Decide consistent terminology (e.g. always call it “Course” not sometimes “Class”, or “Live Class” vs “Lecture”). Consistent naming in the UI and API avoids confusion.Authentication Flow: Sketch how login/registration will work. Plan to use email/password (plus optional OAuth or phone OTP if desired). Plan for password hashing (bcrypt) and tokens (JWT). Use JWT to carry the user’s role so each request carries identity and role.Phase 1 – Architecture & InfrastructureFrontends: Develop three separate frontends (e.g. React/Next.js apps) — one for students, one for teachers, one for admins. Each has its own codebase (or its own directory in a monorepo). On Vercel, create one project per directory so each deploys independently. For example, use a Git monorepo and in Vercel’s dashboard, import your repo three times, each time specifying the root folder (e.g. /student-website, /teacher-website, /admin-website).Routing & Domains: You can host each app on its own subdomain or path. Vercel supports multiple projects under one domain via Microfrontends (using microfrontends.json) or Multi-Zones (using Next.js rewrites). For example, you could assign admin.yourdomain.com to the Admin project, teacher.yourdomain.com to Teacher, etc. Alternatively, use a single Vercel project as a proxy with rewrite rules routing /admin/* to the Admin app and /teacher/* to the Teacher app.Backend: Host your API on Azure App Service (or Azure Functions/Azure Container). Azure App Service fully supports Node/Express apps. Ensure your server listens on the port from process.env.PORT (Azure App Service provides this). For example, in Express use app.listen(process.env.PORT || 3000). Azure’s quickstart docs emphasize this requirement. In Azure setup, create an App Service (Linux) with Node (recommend LTS, e.g. Node 24) and deploy your code (via Azure CLI, Git, or VS Code Azure extension).Database: Choose a managed DB on Azure (e.g. Azure SQL, Azure Database for MySQL/PostgreSQL, or Azure Cosmos DB for MongoDB). For example, Azure Cosmos DB offers a scalable NoSQL (Mongo) or SQL option. Configure your Node app to connect securely (use secrets, SSL).Media Storage: Use Cloudinary to store all uploaded media (images, PDFs, assignment docs, receipts, etc.). Cloudinary’s Node.js SDK lets you upload “images, videos, and any other kind of file” over HTTPS. For instance, when a teacher uploads a PDF, send it to Cloudinary via the SDK/API; store the returned URL (and metadata like filename, uploader, course, date) in your database. Cloudinary handles storage, backup, and optimizations automatically.DNS & CDN: Use Cloudflare for DNS management and CDN protection. Point your domain/subdomains to Cloudflare’s name servers, then create CNAME/A records to your Vercel/ Azure apps. Cloudflare will cache static assets (JS, CSS, images) globally to speed up load times and provides DDoS/WAF protection out-of-the-box. For example, enable Cloudflare’s proxy (orange cloud) on your DNS records so visitors reach Cloudflare’s edge before your servers.Phase 2 – Authentication & Access ControlUser Accounts: Implement signup/login pages in each frontend or centrally. On signup, create a user record (in Users table) with hashed password (e.g. bcrypt), default role “student” or “teacher”/“admin” as appropriate.JWT Tokens: Upon login, authenticate the user (check password), then issue a JWT containing the user’s ID and role. Store this token in a secure cookie or client storage. Each frontend routes and API calls should include this JWT.Backend Checks: In your Node API, add middleware to verify the JWT on each request. Then check the user’s role (from the token) against the required permissions of that route. For example, an endpoint /admin/* should reject requests unless role==="admin". This two-layer protection (frontend route guards + backend checks) ensures students can’t call admin APIs.RBAC Model: Design your backend API routes with RBAC in mind. For example, use Express middleware or decorators to enforce access: only allow “admin” role to call user management endpoints, “teacher” role to upload materials, etc. As TechMacros notes, embed the user’s role in the JWT so “every subsequent request carries the role without an additional database query”. Also keep in mind that authentication (verifying identity) is distinct from authorization (checking role permissions).Account Security: Include “Forgot password” and email confirmation flows. Store an account status (active/blocked/pending) flag in the user record. Implement logout by client-side token removal. Consider session expiration or token refresh policies.Phase 3 – Shared UI & Design SystemComponent Library: Choose or build a consistent design system.
